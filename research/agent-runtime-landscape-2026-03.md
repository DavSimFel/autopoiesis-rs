# Agent Runtime Landscape — March 2026

Updated: 2026-03-22

Method:
- This document uses official product docs and official source repositories only.
- For NVIDIA, I treat **NeMo Agent Toolkit**, **NeMo Guardrails**, and **ACE / Tokkio / ACE Agent** as one layered stack, because NVIDIA currently documents orchestration, safety, and multimodal deployment as separate products rather than one monolithic runtime. ([NeMo Agent Toolkit](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html), [NeMo Guardrails](https://docs.nvidia.com/nemo-guardrails/index.html), [ACE overview](https://docs.nvidia.com/ace/overview/latest/index.html))

## Executive Summary

Comparative note:
- The bullets below are cross-source assessments based on the cited docs, not vendor self-descriptions.

- **OpenClaw** is the broadest "agent platform" in this set: it combines pluggable context engines, markdown-plus-vector memory, multi-agent routing, companion-device nodes, a macOS Canvas/A2UI surface, cron, heartbeats, skills, and plugins inside one gateway. Its own docs also make the trust model explicit: it is a personal-assistant runtime, not a hostile multi-tenant boundary, and its sandboxing docs describe containment as optional and imperfect. ([OpenClaw README](https://raw.githubusercontent.com/openclaw/openclaw/main/README.md), [Context engine](https://docs.openclaw.ai/concepts/context-engine), [Memory](https://docs.openclaw.ai/concepts/memory), [Multi-agent](https://docs.openclaw.ai/concepts/multi-agent), [Nodes](https://docs.openclaw.ai/nodes), [Canvas](https://docs.openclaw.ai/platforms/mac/canvas), [Cron jobs](https://docs.openclaw.ai/automation/cron-jobs), [Heartbeat](https://docs.openclaw.ai/gateway/heartbeat), [Security](https://docs.openclaw.ai/gateway/security), [Sandboxing](https://docs.openclaw.ai/gateway/sandboxing))
- **Claude Code** is the most mature integrated developer experience: terminal, VS Code, JetBrains, Desktop, web/cloud, remote control, subagents, agent teams, CLAUDE.md rules, and auto memory all point to one runtime with strong UX around context, permissions, and collaboration. ([Overview](https://code.claude.com/docs/en/overview), [VS Code](https://code.claude.com/docs/en/vs-code), [Desktop](https://code.claude.com/docs/en/desktop), [Claude Code on the web](https://code.claude.com/docs/en/claude-code-on-the-web), [Subagents](https://code.claude.com/docs/en/sub-agents), [Agent teams](https://code.claude.com/docs/en/agent-teams), [Memory](https://code.claude.com/docs/en/memory), [Permissions](https://code.claude.com/docs/en/permissions), [Sandboxing](https://code.claude.com/docs/en/sandboxing))
- **OpenAI Codex** has the strongest documented local security posture among the coding-first runtimes surveyed: OS-enforced sandboxing, explicit approval policies, protected paths, exec rules, managed proxy/network controls, cloud setup-vs-agent phase separation, and cross-surface config sharing. Its subagent and skills systems are now substantial, but its long-term memory story is still less explicit and less mature in public docs than Claude Code’s or OpenClaw’s. ([Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security), [Sandboxing concepts](https://developers.openai.com/codex/concepts/sandboxing), [Subagents](https://developers.openai.com/codex/subagents), [Skills](https://developers.openai.com/codex/skills), [MCP](https://developers.openai.com/codex/mcp), [Config reference](https://developers.openai.com/codex/config-reference), [AGENTS.md guide](https://developers.openai.com/codex/guides/agents-md), [Changelog](https://developers.openai.com/codex/changelog))
- **NVIDIA’s stack** is the least like the others. It is not a local coding-agent shell runtime; it is an enterprise orchestration and deployment stack. NeMo Agent Toolkit handles workflows, tools, A2A, MCP, observability, and long-term memory integrations; NeMo Guardrails handles programmable safety rails; ACE/Tokkio/ACE Agent handles speech/avatar/microservice deployment. It is strongest on multimodal deployment, protocol interoperability, and enterprise operations, not on local developer-shell containment. ([NeMo Agent Toolkit](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html), [Functions](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/functions-and-function-groups/functions.html), [Memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html), [A2A](https://docs.nvidia.com/nemo/agent-toolkit/latest/components/integrations/a2a.html), [MCP client](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/mcp-client.html), [NeMo Guardrails rail types](https://docs.nvidia.com/nemo/guardrails/latest/about/rail-types.html), [NeMo Guardrails multimodal tutorial](https://docs.nvidia.com/nemo/guardrails/latest/getting-started/tutorials/multimodal.html), [ACE overview](https://docs.nvidia.com/ace/overview/latest/index.html), [ACE Agent overview](https://docs.nvidia.com/ace/ace-agent/4.1/overview.html), [Tokkio components](https://docs.nvidia.com/ace/tokkio/latest/overview/components-and-microservices.html), [Tokkio deployment](https://docs.nvidia.com/ace/tokkio/latest/deployment/deployment.html))

## 1. OpenClaw

### Context Management

- OpenClaw defines context very explicitly: system prompt, conversation history, tool calls/results, and attachments all count toward the window, and `/status`, `/context list`, `/context detail`, and `/compact` are first-class operator surfaces for inspecting that budget. It also counts tool schemas and skill-list metadata as context, not just visible text. ([Context](https://docs.openclaw.ai/concepts/context))
- Older history is compacted into a persisted summary entry in session JSONL, while recent messages stay intact. OpenClaw supports auto-compaction, manual `/compact`, strict identifier preservation during summarization, and even a separate compaction model from the main model. ([Compaction](https://docs.openclaw.ai/concepts/compaction))
- OpenClaw distinguishes **compaction** from **session pruning**. Compaction rewrites history with a persistent summary; pruning only trims old tool results from the in-memory prompt, and is specifically documented as an Anthropic prompt-cache optimization path. ([Compaction](https://docs.openclaw.ai/concepts/compaction), [Session pruning](https://docs.openclaw.ai/concepts/session-pruning))
- Context assembly is pluggable. A context-engine plugin can participate in `ingest`, `assemble`, `compact`, and `afterTurn`, inject extra system prompt text, and optionally fully own compaction behavior. That is a deeper context abstraction than most coding runtimes expose publicly. ([Context engine](https://docs.openclaw.ai/concepts/context-engine))

### Safety / Sandboxing

- OpenClaw’s sandboxing is optional and tool-focused: the gateway remains on the host, while tool execution can be sandboxed with Docker, SSH, or OpenShell backends. Modes include `off`, `non-main`, and `all`; scopes include `session`, `agent`, and `shared`; workspace access can be `none`, `ro`, or `rw`. ([Sandboxing](https://docs.openclaw.ai/gateway/sandboxing))
- The sandbox docs are careful not to oversell containment. They say sandboxing "is not a perfect security boundary" and note that elevated exec can bypass sandboxing entirely. ([Sandboxing](https://docs.openclaw.ai/gateway/sandboxing))
- The security docs also narrow the trust model: one trusted operator boundary per gateway, explicitly not a supported hostile multi-tenant environment. That is strategically important because OpenClaw’s feature set can otherwise look more isolated than the docs actually claim. ([Security](https://docs.openclaw.ai/gateway/security))
- Session tools have their own visibility controls, and sandboxed sessions are clamped to spawned-session visibility by default. That is a notable attempt to keep subagents from freely browsing sibling sessions even inside the same gateway. ([Session tool](https://docs.openclaw.ai/concepts/session-tool))

### Multi-Agent

- OpenClaw’s "one gateway, many agents" model is strong. Each agent gets its own workspace, `agentDir`, session store, and auth profiles, while inbound channel/account/peer bindings deterministically route messages to the correct agent. ([Multi-agent](https://docs.openclaw.ai/concepts/multi-agent))
- OpenClaw documents four cross-session tools: `sessions_list`, `sessions_history`, `sessions_send`, and `sessions_spawn`. `sessions_send` can do routed inter-session messaging plus reply-back loops and announce steps; `sessions_spawn` creates isolated subagent sessions and posts results back to the requester channel. ([Session tool](https://docs.openclaw.ai/concepts/session-tool))
- Subagent spawning has allowlists, sandbox inheritance rules, thread-binding options, and post-run archive policies. OpenClaw’s docs also state that subagents default to the full tool set minus the session tools, and subagents cannot spawn further subagents. ([Session tool](https://docs.openclaw.ai/concepts/session-tool))
- The runtime also supports agent-to-agent messaging across isolated agents inside one gateway and across channel accounts, which is conceptually closer to "message-routed sessions" than to a planner/worker tree. ([OpenClaw README](https://raw.githubusercontent.com/openclaw/openclaw/main/README.md), [Multi-agent](https://docs.openclaw.ai/concepts/multi-agent))

### Memory

- OpenClaw’s documented source of truth for memory is plain Markdown in the workspace: daily append-only logs under `memory/YYYY-MM-DD.md` plus optional curated `MEMORY.md`. The model only "remembers" what gets written to disk. ([Memory](https://docs.openclaw.ai/concepts/memory))
- It exposes `memory_search` and `memory_get` tools, and documents vector search over `MEMORY.md` and `memory/*.md`, with hybrid search, multiple embedding providers, optional QMD sidecar support, and extra ranking features referenced from the memory configuration docs. ([Memory](https://docs.openclaw.ai/concepts/memory))
- Before auto-compaction, OpenClaw can run a silent "memory flush" turn that nudges the model to write durable notes to disk before summarizing older context away. That is a clever bridge between transient context pressure and long-term memory persistence. ([Memory](https://docs.openclaw.ai/concepts/memory), [Compaction](https://docs.openclaw.ai/concepts/compaction))

### Unique Features

- **Nodes:** OpenClaw can pair companion devices over the gateway WebSocket and expose command families like `canvas.*`, `camera.*`, `screen.record`, `location.get`, `notifications.*`, `device.*`, and `system.run` via `node.invoke`. That is a richer device/peripheral story than the coding-agent runtimes document. ([Nodes](https://docs.openclaw.ai/nodes))
- **Canvas:** the macOS app hosts a controllable `WKWebView` canvas with a custom `openclaw-canvas://` scheme, live reload, JS eval, snapshots, and A2UI support. ([Canvas](https://docs.openclaw.ai/platforms/mac/canvas))
- **Heartbeat + cron:** OpenClaw supports periodic heartbeats in the main session, optional light-context and isolated-session heartbeats, plus persisted cron jobs that can target main/current/custom/isolated sessions and deliver via channel announce or webhook. ([Heartbeat](https://docs.openclaw.ai/gateway/heartbeat), [Cron jobs](https://docs.openclaw.ai/automation/cron-jobs))
- **Plugins + bundles:** native plugins can register providers, channels, tools, speech, web search, services, and context engines, and OpenClaw also recognizes Codex/Claude/Cursor-compatible plugin bundle layouts. That cross-runtime bundle compatibility is unusual. ([Plugins](https://docs.openclaw.ai/tools/plugin), [Plugin manifest](https://docs.openclaw.ai/plugins/manifest))
- **Skills distribution:** OpenClaw ships bundled skills, supports local/workspace/plugin skill layers, and has a public ClawHub registry. ([Skills](https://docs.openclaw.ai/tools/skills), [OpenClaw README](https://raw.githubusercontent.com/openclaw/openclaw/main/README.md))

## 2. Claude Code / "Claude Cowork"

Note:
- Anthropic’s public docs describe **Claude Code** across terminal, IDE, desktop, web/cloud, remote-control, and team collaboration surfaces. I did not find a separately documented runtime named "Claude Cowork", so I treat "Cowork" here as the collaborative/cloud surfaces around Claude Code. ([Overview](https://code.claude.com/docs/en/overview), [Claude Code on the web](https://code.claude.com/docs/en/claude-code-on-the-web), [Desktop](https://code.claude.com/docs/en/desktop), [Remote control](https://code.claude.com/docs/en/remote-control))

### Context Management

- Claude Code starts every session with a fresh context window, then carries knowledge across sessions via two documented layers: `CLAUDE.md` files and auto memory. `CLAUDE.md` can be managed, project-scoped, or user-scoped; `.claude/rules/` can scope instructions to paths. ([Memory](https://code.claude.com/docs/en/memory))
- Auto memory stores project-specific notes under `~/.claude/projects/<project>/memory/`, loads the first 200 lines of `MEMORY.md` at the start of every session, and shares one memory directory across all worktrees and subdirectories inside the same Git repo. ([Memory](https://code.claude.com/docs/en/memory))
- Claude Code exposes user-facing context controls that are unusually mature: `/compact` for conversation compaction, `/context` for visual context usage plus optimization hints, and `/btw` for tool-less ephemeral side questions that can see the full current conversation without entering history. ([Commands](https://code.claude.com/docs/en/commands), [Interactive mode](https://code.claude.com/docs/en/interactive-mode))
- The task list persists across context compactions. That is a subtle but important difference from runtimes that treat compaction as just transcript shortening. ([Interactive mode](https://code.claude.com/docs/en/interactive-mode))

### Safety / Sandboxing

- Claude Code has a deny-first permission model: rules evaluate `deny -> ask -> allow`, with permission modes `default`, `acceptEdits`, `plan`, `dontAsk`, and `bypassPermissions`. ([Permissions](https://code.claude.com/docs/en/permissions))
- Sandboxing is OS-level and specifically applies to the Bash tool and its subprocesses: Seatbelt on macOS, bubblewrap on Linux/WSL2. Filesystem and network restrictions are enforced at the OS level, while the permission system still applies to all tools. ([Sandboxing](https://code.claude.com/docs/en/sandboxing), [Permissions](https://code.claude.com/docs/en/permissions))
- Anthropic also exposes managed settings for enterprises: disabling bypass mode, forcing managed-only permission rules or hooks, constraining MCP servers, and constraining allowed sandbox domains and read paths. ([Permissions](https://code.claude.com/docs/en/permissions))
- An important limitation: the sandbox is a Bash sandbox, not a universal runtime sandbox for every tool. That is still strong, but less universal than Codex’s "spawned commands inherit the same boundary" framing across more of the runtime. ([Sandboxing](https://code.claude.com/docs/en/sandboxing))

### Multi-Agent

- The built-in `Agent` tool spawns subagents with their own context windows. Anthropic documents built-in Explore, Plan, and general-purpose agents, plus helper agents, and explicitly positions subagents as a way to reduce context pollution in the main conversation. ([Tools reference](https://code.claude.com/docs/en/tools-reference), [Subagents](https://code.claude.com/docs/en/sub-agents))
- Custom subagents are unusually configurable: tools/disallowedTools, model, permissionMode, hooks, `mcpServers`, `skills`, `memory`, `background`, `effort`, and `isolation = worktree`. ([Subagents](https://code.claude.com/docs/en/sub-agents))
- The main thread can restrict which subagent types it may spawn using `Agent(agent_type)` syntax. That is a nice safety/cost control that most agent runtimes do not document this explicitly. ([Subagents](https://code.claude.com/docs/en/sub-agents))
- Anthropic’s **agent teams** go beyond subagents. Teams have a lead, separate teammate sessions, shared task lists, mailboxes, direct teammate messaging, and either in-process or split-pane display modes. Tasks use file locking for claims, and team/task state is stored locally under `~/.claude/teams/` and `~/.claude/tasks/`. ([Agent teams](https://code.claude.com/docs/en/agent-teams))

### Memory

- Claude Code’s memory model is the clearest public one in the survey: `CLAUDE.md` for user-authored rules, auto memory for learned notes, explicit `/memory` tooling, and stable file locations/scopes. ([Memory](https://code.claude.com/docs/en/memory), [Commands](https://code.claude.com/docs/en/commands))
- Subagents can also have persistent memory in `user`, `project`, or `local` scope, with the first 200 lines of their memory `MEMORY.md` injected and read/write tools automatically enabled when memory is on. ([Subagents](https://code.claude.com/docs/en/sub-agents))
- Anthropic explicitly says `CLAUDE.md` is context, not hard enforcement. That distinction matters: Claude Code separates instruction memory from managed technical enforcement in settings. ([Memory](https://code.claude.com/docs/en/memory))

### Unique Features

- **Cross-surface cohesion:** Claude Code’s docs present one runtime across terminal, VS Code, JetBrains, Desktop, the web, mobile monitoring, and remote control. Tasks can start locally and continue in the cloud or vice versa. ([Overview](https://code.claude.com/docs/en/overview), [VS Code](https://code.claude.com/docs/en/vs-code), [Desktop](https://code.claude.com/docs/en/desktop), [Claude Code on the web](https://code.claude.com/docs/en/claude-code-on-the-web))
- **IDE-native workflow:** the VS Code extension supports inline plan review, file/line-range `@` mentions, multiple conversations, checkpoints, browser automation via Chrome, and a shared config/memory/MCP story with the CLI. ([VS Code](https://code.claude.com/docs/en/vs-code))
- **Desktop surfaces:** the Desktop app adds live app preview, diff comments, GitHub PR monitoring, local/SSH/cloud environments, connectors, parallel worktree-isolated sessions, and recurring scheduled tasks. ([Desktop](https://code.claude.com/docs/en/desktop))
- **Plugin packaging:** Claude plugins can package skills, agents, hooks, MCP servers, and LSP servers, and can be distributed through marketplaces. ([Plugins](https://code.claude.com/docs/en/plugins))

## 3. OpenAI Codex

### Context Management

- Codex builds its instruction chain from global and project `AGENTS.md` layers before starting work. It reads a global file under `CODEX_HOME`, then project files from repo root down to the current directory, and caps the combined size at `project_doc_max_bytes` (32 KiB by default). ([AGENTS.md guide](https://developers.openai.com/codex/guides/agents-md))
- Codex persists transcript history through `history.persistence` and `history.max_bytes`, storing history in `history.jsonl` under `CODEX_HOME` when enabled. ([Config reference](https://developers.openai.com/codex/config-reference), [Advanced config](https://developers.openai.com/codex/config-advanced))
- Public config keys clearly show a compaction system: `compact_prompt`, `experimental_compact_prompt_file`, and `model_auto_compact_token_limit`, and Codex emits `task.compact` metrics. ([Config reference](https://developers.openai.com/codex/config-reference), [Advanced config](https://developers.openai.com/codex/config-advanced))
- Assessment: Codex clearly supports automatic/local compaction, but its public UX and persistence semantics are less explicitly documented than Claude Code’s `/compact` flow or OpenClaw’s "summary persisted to session JSONL" docs. This is an inference from the current public docs, not an absence proof. ([Config reference](https://developers.openai.com/codex/config-reference), [Advanced config](https://developers.openai.com/codex/config-advanced), [Changelog](https://developers.openai.com/codex/changelog))

### Safety / Sandboxing

- Codex’s safety model is explicit and layered: sandbox mode defines what commands can technically do, approval policy defines when Codex must stop and ask. Defaults are network off plus an OS-enforced workspace sandbox locally, and a setup-phase/agent-phase split in Codex cloud. ([Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security), [Sandboxing concepts](https://developers.openai.com/codex/concepts/sandboxing))
- Local sandbox modes are `read-only`, `workspace-write`, and `danger-full-access`; approval policies are `untrusted`, `on-request`, `never`, or granular category controls. ([Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security), [Config reference](https://developers.openai.com/codex/config-reference))
- Protected paths are documented inside writable roots: `.git`, `.agents`, and `.codex` remain read-only recursively even in `workspace-write`. ([Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security))
- Platform enforcement is detailed: Seatbelt on macOS, bubblewrap plus seccomp on Linux, WSL/Linux semantics for Windows when configured, and a native Windows sandbox path otherwise. ([Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security), [Sandboxing concepts](https://developers.openai.com/codex/concepts/sandboxing))
- Codex also exposes a separate **exec policy** layer: Starlark `prefix_rule()` rules can allow, prompt, or forbid command prefixes outside the sandbox, and the "most restrictive decision wins." ([Exec policy](https://developers.openai.com/codex/exec-policy))
- The cloud model is stronger than most competitors’ public docs: setup runs with network and secrets, then the agent phase runs offline by default, and secrets are removed before the agent phase. ([Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security))

### Multi-Agent

- Codex subagent workflows are now first-class and enabled by default, but Codex only spawns them when explicitly asked. Built-in agent types are `default`, `worker`, and `explorer`; custom agents live in `~/.codex/agents/` or `.codex/agents/`. ([Subagents](https://developers.openai.com/codex/subagents), [Config reference](https://developers.openai.com/codex/config-reference))
- Subagents inherit the parent sandbox policy and live approval/runtime overrides unless explicitly overridden, which keeps the security model coherent across the tree. ([Subagents](https://developers.openai.com/codex/subagents), [Changelog](https://developers.openai.com/codex/changelog))
- The multi-agent system includes spawn/send/wait/close orchestration, custom per-agent configs, limits like `agents.max_threads` and `agents.max_depth`, and a batch fan-out path through `spawn_agents_on_csv`. ([Subagents](https://developers.openai.com/codex/subagents), [Config reference](https://developers.openai.com/codex/config-reference))
- Codex also has an experimental "smart approvals" path that can route approval requests through a guardian reviewer subagent. ([Config reference](https://developers.openai.com/codex/config-reference), [Changelog](https://developers.openai.com/codex/changelog))

### Memory

- Codex’s clearly documented persistent layers today are: `AGENTS.md` instruction chains, skills, project/user/admin/system skill directories, project/user config layers, and optional transcript history persistence. ([AGENTS.md guide](https://developers.openai.com/codex/guides/agents-md), [Skills](https://developers.openai.com/codex/skills), [Config reference](https://developers.openai.com/codex/config-reference), [Advanced config](https://developers.openai.com/codex/config-advanced))
- Assessment: OpenAI has not yet documented a stable end-user memory system comparable to Claude Code’s auto memory or OpenClaw’s markdown-plus-vector memory. The changelog does show active memory work, including memory citations and memory-related behavior changes, so this area appears to be evolving rather than absent. ([Changelog](https://developers.openai.com/codex/changelog), [Config reference](https://developers.openai.com/codex/config-reference))

### Unique Features

- **Security-first local runtime:** among the coding-first runtimes surveyed, Codex most thoroughly documents sandbox modes, approval modes, protected paths, managed network proxy behavior, granular approval categories, exec policy rules, and cloud phase separation. ([Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security), [Sandboxing concepts](https://developers.openai.com/codex/concepts/sandboxing), [Exec policy](https://developers.openai.com/codex/exec-policy))
- **Skills standard:** Codex uses the open agent skills standard, progressive disclosure, multi-scope skill discovery, explicit `agents/openai.yaml` dependency metadata, and ships built-in `skill-creator` and `skill-installer`. ([Skills](https://developers.openai.com/codex/skills))
- **Cross-surface consistency:** CLI, IDE extension, and Codex app share MCP configuration and skills; the app adds parallel threads, worktrees, automations, integrated terminal, and Git workflows. ([MCP](https://developers.openai.com/codex/mcp), [App features](https://developers.openai.com/codex/app/features), [IDE features](https://developers.openai.com/codex/ide/features))
- **Realtime/audio plumbing:** current Codex releases added realtime startup context, transcription mode, and audio-playback handling, but the coding models themselves are still documented as text/image input and text output only, while `gpt-realtime-1.5` is the actual audio-in/audio-out model. ([Changelog](https://developers.openai.com/codex/changelog), [GPT-5.3-Codex model page](https://developers.openai.com/api/docs/models/gpt-5.3-codex), [gpt-realtime-1.5 model page](https://developers.openai.com/api/docs/models/gpt-realtime-1.5))

## 4. NVIDIA ACE / NeMo Agent

### Context Management

- NeMo Agent Toolkit centers context management more around workflows and long-term memory backends than around transcript compaction UX. It exposes a dedicated memory subsystem for conversation history, preferences, and long-term memory, plus an `auto_memory_agent` wrapper that guarantees capture and retrieval without requiring the model to explicitly call memory tools. ([NeMo Agent Toolkit memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html))
- NAT includes a web UI with streaming responses, chat history, conversation management, intermediate-step visualization, and human-in-the-loop support, but its docs do not foreground a `/compact`-style operator surface or transcript compaction system the way Claude Code or OpenClaw do. This is an inference from the current public docs. ([UI](https://docs.nvidia.com/nemo/agent-toolkit/latest/run-workflows/launching-ui.html), [NeMo Agent Toolkit memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html))
- NAT’s long-term memory backends are pluggable: Mem0, Redis, and Zep are included via plugins, and the wrapper agent supports multi-tenant isolation by deriving a user id at runtime. ([NeMo Agent Toolkit memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html))

### Safety / Sandboxing

- NVIDIA’s safety story is primarily **Guardrails**, not host sandboxing. NeMo Guardrails inserts programmable rails at input, retrieval, dialog, execution/tool, and output stages. Built-ins cover jailbreak detection, topic control, PII handling, fact checking, hallucination detection, and injection detection. ([Rail types](https://docs.nvidia.com/nemo/guardrails/latest/about/rail-types.html), [Configuration reference](https://docs.nvidia.com/nemo/guardrails/latest/configure-rails/configuration-reference.html))
- Guardrails also supports multimodal safety for text+image inputs and outputs, using vision models as judges. ([Multimodal tutorial](https://docs.nvidia.com/nemo/guardrails/latest/getting-started/tutorials/multimodal.html))
- NAT’s A2A server can be protected with OAuth2 and JWT validation, and NAT’s MCP client supports auth providers for protected MCP servers. ([A2A server](https://docs.nvidia.com/nemo/agent-toolkit/latest/run-workflows/a2a-server.html), [MCP client](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/mcp-client.html))
- Assessment: compared with Codex or Claude Code, NVIDIA’s docs do not describe a comparable OS-level shell sandbox for model-issued commands on a developer machine. The safety model is policy/rail/deployment-centric rather than local shell-containment-centric. ([NeMo Guardrails](https://docs.nvidia.com/nemo-guardrails/index.html), [Rail types](https://docs.nvidia.com/nemo/guardrails/latest/about/rail-types.html), [ACE Agent overview](https://docs.nvidia.com/ace/ace-agent/4.1/overview.html))

### Multi-Agent

- In NAT, agents and tools are both functions, so an agent can be used as a tool by another agent. NVIDIA explicitly points to this as the basis for multi-agent workflows. ([Functions](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/functions-and-function-groups/functions.html), [Agents](https://docs.nvidia.com/nemo/agent-toolkit/latest/components/agents/index.html))
- NAT includes multiple built-in agent/workflow types, including ReAct, ReWOO, router agents, sequential executors, tool-calling agents, and an automatic memory wrapper. ([Agents](https://docs.nvidia.com/nemo/agent-toolkit/latest/components/agents/index.html))
- NVIDIA has gone hard on protocol interop: NAT can be an MCP client and an MCP or FastMCP server, and it can also be an A2A client and A2A server. A2A support includes Agent Cards, skill discovery, remote delegation, and protocol-compliant publication of workflows as agents. ([NeMo Agent Toolkit index](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html), [MCP client](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/mcp-client.html), [A2A](https://docs.nvidia.com/nemo/agent-toolkit/latest/components/integrations/a2a.html), [A2A server](https://docs.nvidia.com/nemo/agent-toolkit/latest/run-workflows/a2a-server.html))
- NAT is explicitly framework-agnostic and designed to wrap or connect existing agent systems such as LangChain, LlamaIndex, CrewAI, Semantic Kernel, and Google ADK. ([NeMo Agent Toolkit index](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html))

### Memory

- NAT’s documented memory system is stronger than its transcript-management story. The `auto_memory_agent` wrapper guarantees save/retrieve behavior and is backend-agnostic. ([NeMo Agent Toolkit memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html))
- ACE Agent itself also supports multi-turn and multi-user contextual conversation flow, but NVIDIA’s public docs frame that more as dialog modeling plus deployment than as user-facing memory tooling. ([ACE Agent overview](https://docs.nvidia.com/ace/ace-agent/4.1/overview.html))

### Unique Features

- **Multimodal deployment:** ACE is built for real-time avatars and digital humans, integrating speech, translation, voice transfer, emotion, animation, rendering, and microservice orchestration. ([ACE overview](https://docs.nvidia.com/ace/overview/latest/index.html), [Tokkio components](https://docs.nvidia.com/ace/tokkio/latest/overview/components-and-microservices.html))
- **Enterprise deployment model:** ACE/Tokkio documents Helm/Kubernetes deployment, controller and application instances, infrastructure scripts, TURN/media relay, and multi-cloud/bare-metal options. ([Tokkio deployment](https://docs.nvidia.com/ace/tokkio/latest/deployment/deployment.html))
- **Observability / evaluation / profiling:** NAT includes profiler support, built-in evaluation, OpenTelemetry compatibility, and integrations with Phoenix, Weave, and Langfuse. ([NeMo Agent Toolkit index](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html))
- **Dialog modeling via Colang:** NVIDIA positions Colang as especially useful when you need multimodal experiences, input/output guardrails, and granular flow control. ([Planning your bot](https://docs.nvidia.com/ace/ace-agent/4.1/best-practices/planning-bot.html))

### What autopoiesis-rs Vision Is Missing, Compared With Competitors

Assessment:
- **A first-class surface strategy is missing from the vision.** The vision is strong on queue, shell, subscriptions, and CLI self-management, but it does not currently articulate an IDE surface, a desktop/app surface, or a cloud/offload workflow. Claude Code and Codex both treat those as first-class, and OpenClaw/NVIDIA both have companion app or deployment surfaces. ([Vision](../docs/vision.md), [VS Code](https://code.claude.com/docs/en/vs-code), [Desktop](https://code.claude.com/docs/en/desktop), [Claude Code on the web](https://code.claude.com/docs/en/claude-code-on-the-web), [IDE features](https://developers.openai.com/codex/ide/features), [App features](https://developers.openai.com/codex/app/features), [Canvas](https://docs.openclaw.ai/platforms/mac/canvas), [Tokkio deployment](https://docs.nvidia.com/ace/tokkio/latest/deployment/deployment.html))
- **The vision does not yet define an ecosystem/interoperability position.** Competitors are converging around MCP, A2A, plugin bundles, skills packaging, and marketplaces. Autopoiesis says "one tool" and "shell is the universal tool", but it does not yet explain how it will participate in that ecosystem without compromising the one-tool principle. ([Vision](../docs/vision.md), [Plugins](https://docs.openclaw.ai/tools/plugin), [Plugins](https://code.claude.com/docs/en/plugins), [Skills](https://developers.openai.com/codex/skills), [MCP](https://developers.openai.com/codex/mcp), [NeMo Agent Toolkit index](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html), [A2A](https://docs.nvidia.com/nemo/agent-toolkit/latest/components/integrations/a2a.html))
- **Long-term memory is under-specified in the vision compared with the market.** The vision is unusually strong on explicit context loading via subscriptions, but it says much less about durable memory representation, recall policy, curation, and promotion from transcript to memory than OpenClaw, Claude Code, or NAT do. ([Vision](../docs/vision.md), [Memory](https://docs.openclaw.ai/concepts/memory), [Memory](https://code.claude.com/docs/en/memory), [NeMo Agent Toolkit memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html))
- **Parallel coding isolation is under-specified.** Competitors increasingly formalize worktrees, isolated agents, or isolated background sessions. The autopoiesis vision talks about sessions and agent-to-agent messaging, but it does not yet define a branch/worktree model for parallel code changes. ([Vision](../docs/vision.md), [Desktop](https://code.claude.com/docs/en/desktop), [Subagents](https://code.claude.com/docs/en/sub-agents), [App features](https://developers.openai.com/codex/app/features), [Session tool](https://docs.openclaw.ai/concepts/session-tool))
- **Observability and evaluation are not prominent in the vision.** NVIDIA in particular treats profiling, observability, and evaluation as core runtime features. The autopoiesis vision emphasizes auditability and SQLite state, but does not yet present a comparable evaluation or tracing story. ([Vision](../docs/vision.md), [NeMo Agent Toolkit index](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html))
- **The multimodal / companion-device question is unresolved.** OpenClaw and NVIDIA both show real strategic value in nodes, Canvas, voice, and avatar/device channels. The autopoiesis vision currently stays text-and-shell centric. That is fine if intentional, but the vision should say so explicitly. ([Vision](../docs/vision.md), [Nodes](https://docs.openclaw.ai/nodes), [Canvas](https://docs.openclaw.ai/platforms/mac/canvas), [ACE overview](https://docs.nvidia.com/ace/overview/latest/index.html))

### What autopoiesis-rs Vision Has That No Competitor Currently Matches

Assessment:
- **A single universal tool surface.** The vision insists on one tool, the shell, with the prompt teaching usage. Every competitor surveyed exposes a larger native tool/function/app surface or a protocol/plugin runtime. I did not find another runtime that makes "one tool only" this central. ([Vision](../docs/vision.md), [Tools reference](https://code.claude.com/docs/en/tools-reference), [Plugins](https://docs.openclaw.ai/tools/plugin), [Config reference](https://developers.openai.com/codex/config-reference), [Functions](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/functions-and-function-groups/functions.html))
- **File-only subscriptions as primary context management.** The vision’s core mechanism is not generic memory or retrieval; it is explicit file subscriptions with line/range/regex/head-tail/jq filters and timestamp-based placement in history. None of the surveyed runtimes documents an equivalent file-subscription layer as the main context primitive. ([Vision](../docs/vision.md), [Context](https://docs.openclaw.ai/concepts/context), [Memory](https://code.claude.com/docs/en/memory), [AGENTS.md guide](https://developers.openai.com/codex/guides/agents-md), [NeMo Agent Toolkit memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html))
- **Shell-output capping with forced file-backed rereads.** The vision intentionally caps inline shell output, stores the full result in a file, and forces the agent to subscribe to that file to re-read it. That is a very opinionated anti-context-bloat mechanism I did not find documented elsewhere. ([Vision](../docs/vision.md))
- **One inbox / one queue for everything.** The vision puts cron, webhook, user, and agent messages into one SQLite queue and drives all work through the same agent loop. Other runtimes have schedulers and background jobs, but not this explicit "single inbox for all sources" architectural law. ([Vision](../docs/vision.md))
- **Layered identity with chain-of-command metadata and planned taint-aware safety.** The combination of `constitution.md`, `operator.md`, self-modifiable `identity.md`, `context.md`, message principals, standing approvals, taint tracking, and budgets is more explicit and more coherent than the public safety/identity narratives in the surveyed runtimes. ([Vision](../docs/vision.md), [Roadmap](../docs/roadmap.md))
- **Topics as portable context indexes.** The vision treats topics as optional indexes over subscriptions, with export/import semantics, rather than as containers or opaque memory blobs. That is strategically distinct from the more common "agent memory" or "context compaction" framing. ([Vision](../docs/vision.md))

### Concrete Recommendations For Vision Updates

- **Keep the one-tool principle, but add a protocol/interoperability layer explicitly.** Recommendation: state that MCP, A2A, webhooks, and future connectors are ingress/egress adapters managed through shell and queue semantics, not new core tools. That preserves the architecture while acknowledging where the ecosystem is converging. ([Vision](../docs/vision.md), [MCP](https://developers.openai.com/codex/mcp), [A2A](https://docs.nvidia.com/nemo/agent-toolkit/latest/components/integrations/a2a.html), [Plugins](https://docs.openclaw.ai/tools/plugin))
- **Promote subscriptions to the flagship differentiator.** Recommendation: sharpen the vision copy around subscriptions as a first-class alternative to opaque memory systems, and define the operator UX for freshness, budget, provenance, and topic snapshots/export. This is the part of the vision that is most clearly non-me-too. ([Vision](../docs/vision.md))
- **Add a first-class surface strategy.** Recommendation: define a three-layer surface plan: CLI first, IDE companion second, remote worker/cloud executor third, all using the same queue/turn architecture. Competitors have made this table-stakes for serious agent workflows. ([Vision](../docs/vision.md), [VS Code](https://code.claude.com/docs/en/vs-code), [App features](https://developers.openai.com/codex/app/features), [Claude Code on the web](https://code.claude.com/docs/en/claude-code-on-the-web))
- **Define durable memory separately from subscriptions.** Recommendation: keep subscriptions as context loading, but add a separate durable-memory model: where durable facts live, how they are written, how they are cited, and how they are promoted or pruned. OpenClaw, Claude Code, and NAT all show that users expect this to be explicit. ([Vision](../docs/vision.md), [Memory](https://docs.openclaw.ai/concepts/memory), [Memory](https://code.claude.com/docs/en/memory), [NeMo Agent Toolkit memory](https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html))
- **Add an isolation story for parallel coding work.** Recommendation: define whether autopoiesis will use worktrees, ephemeral child workspaces, or something else when multiple agents touch code in parallel. This matters for both product UX and safety. ([Vision](../docs/vision.md), [Desktop](https://code.claude.com/docs/en/desktop), [App features](https://developers.openai.com/codex/app/features), [Session tool](https://docs.openclaw.ai/concepts/session-tool))
- **Add an explicit multimodal stance.** Recommendation: either say "text + shell first; device/multimodal is out of scope" or define a node/companion-device model. Leaving it unstated makes strategic positioning ambiguous, especially given OpenClaw’s nodes and NVIDIA’s ACE. ([Vision](../docs/vision.md), [Nodes](https://docs.openclaw.ai/nodes), [ACE overview](https://docs.nvidia.com/ace/overview/latest/index.html))
- **Add an observability/evaluation chapter to the vision.** Recommendation: define what must be traceable, auditable, benchmarked, and replayable for autonomous behavior. NVIDIA’s stack shows how strong enterprise buyers now expect this story to be. ([Vision](../docs/vision.md), [NeMo Agent Toolkit index](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html))
- **Keep the safety section, but make the operator-facing modes concrete.** Recommendation: add named permission presets and escalation modes to the vision, not just abstract dimensions. Codex and Claude Code both benefit from operator-comprehensible presets. ([Vision](../docs/vision.md), [Permissions](https://code.claude.com/docs/en/permissions), [Agent approvals & security](https://developers.openai.com/codex/agent-approvals-security))

## Current autopoiesis-rs Implementation Note

This is not a competitor comparison point; it is a strategic normalization note for this repo:

- The current codebase is explicitly not ready to make strong autonomy or security claims. The repo’s own risk document says shell containment is not meaningful today, HTTP callers can inject arbitrary message roles, denial does not terminate turns, queue claiming is not atomic, and identity immutability is not yet enforced. ([Current risks](../docs/current/risks.md))
- The roadmap already reflects the right sequencing: P0 fixes first, then standing approvals, taint tracking, budget enforcement, subscription system, context assembly, topics, triggers, and only later heavier sandboxing. Strategic product work should not jump ahead of that ordering. ([Roadmap](../docs/roadmap.md), [Current risks](../docs/current/risks.md))

## Bottom Line

- If autopoiesis-rs wants to compete on **developer UX breadth**, Claude Code and Codex are the benchmarks. ([Overview](https://code.claude.com/docs/en/overview), [README](https://raw.githubusercontent.com/openai/codex/main/README.md))
- If it wants to compete on **personal-assistant platform breadth**, OpenClaw is the closest comparator. ([OpenClaw README](https://raw.githubusercontent.com/openclaw/openclaw/main/README.md))
- If it wants to compete on **enterprise orchestration / multimodal deployment**, NVIDIA’s stack is the benchmark, but it is a different category of product. ([NeMo Agent Toolkit](https://docs.nvidia.com/nemo/agent-toolkit/latest/index.html), [ACE overview](https://docs.nvidia.com/ace/overview/latest/index.html))
- The clearest strategic lane for autopoiesis-rs is not "another MCP/plugin coding agent." It is: **one queue, one shell tool, explicit file subscriptions, durable chain-of-command identity, and taint-aware autonomy**. That is the part of the vision that currently looks most differentiated. ([Vision](../docs/vision.md), [Roadmap](../docs/roadmap.md))

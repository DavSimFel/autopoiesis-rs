- P1: [src/agent/loop_impl.rs/token_total + charged_turn_meta] The partial-usage fix now undercounts instead of overcounting. When the provider returns only `input_tokens` or only `output_tokens`, the budget gate and persisted turn meta treat the missing side as `0`, so a prompt-heavy or response-heavy turn can slip past `max_tokens_per_turn`/session/day enforcement. The new helper tests lock in that undercounted behavior rather than catching it.
- P1: [src/agent/queue.rs, src/main.rs, src/agent/spawn.rs/drain result semantics] `drain_queue_with_stats()` still returns the first denial even after a later queued user row executes successfully. The new completion test proves the parent notification now happens, but CLI drains still print a denial for a queue that also ran a later success, and spawned-child drains still error out early in `finish_spawned_child_drain()` before applying the successful child result.
- P2: [src/agent/loop_impl.rs/persisted_inbound_user_message call sites] `persisted_inbound_user_message()` now always falls back to `Some(original_user_message.clone())`, so every `ok_or_else(\"missing user message after inbound checks\")` branch is dead. That is low risk, but it hides the actual invariant and makes future callers look fallible when they are not.
